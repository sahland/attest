# a11y_audit — спецификация стартового набора правил (MVP)

Библиотека непрерывного соответствия доступности для Flutter. Этот документ — build-ready ТЗ на первый набор правил: что детектим, как детектим в дереве семантики, к какому пункту стандарта привязываем и насколько проверка автоматизируема.

Стандарт-ориентир: **EN 301 549 v3.2.1** (= WCAG 2.1 AA в полном объёме) с заделом под **WCAG 2.2 / EN v4.1.1**. Правила версионируются в паки, чтобы переезд на новый стандарт не ломал пользователей.

---

## 1. Принципы, на которых стоит весь движок

1. **Источник истины — дерево семантики Flutter (`SemanticsNode`).** Это ровно то, что потребляют TalkBack и VoiceOver. Если узел невидим для дерева семантики — он невидим для скринридера. Поэтому большинство проверок — это обход `SemanticsNode` в обычном widget-тесте, без реального девайса.

2. **Каждая находка трассируется к пункту стандарта.** Не «контраст плохой», а «WCAG 1.4.3 / EN 301 549 §11.5.2.4, контраст 3.1:1, требуется ≥4.5:1, файл X строка Y». Это и есть то, что отличает compliance-инструмент от линтера.

3. **Честность об автоматизируемости.** Каждое правило помечено уровнем достоверности. Автоматика ловит ~30–40% проблем доступности; остальное — ручная проверка. Инструмент НЕ обещает «соответствие», он обещает «автоматическое покрытие + структурированный чеклист для ручной части».

---

## 2. Модель данных

```dart
/// Одна находка аудита.
class Finding {
  final String ruleId;            // 'a11y/interactive-name'
  final Criterion criterion;      // WCAG SC + пункт EN 301 549
  final Severity severity;        // error | warning | info
  final Confidence confidence;    // deterministic | heuristic
  final SourceLocation? location; // файл:строка (через debugCreator узла)
  final Rect bounds;              // глобальный прямоугольник элемента
  final String message;           // человекочитаемое описание
  final String suggestion;        // как починить (с примером кода)
  final String fingerprint;       // стабильный хеш для baseline-сравнения
}

class Criterion {
  final String wcag;        // '1.4.3'
  final String wcagLevel;   // 'A' | 'AA'
  final String en301549;    // '11.5.2.4'
  final String title;       // 'Contrast (Minimum)'
}

enum Severity { error, warning, info }
enum Confidence { deterministic, heuristic }
```

`fingerprint` — критичная штука для baseline-гейта. Он должен быть стабилен между прогонами (не зависеть от случайного порядка обхода), но меняться при настоящей регрессии. Рекомендуемый состав: `ruleId` + нормализованный путь виджета в дереве (по типам, не по hashCode) + `criterion.wcag`. Координаты в fingerprint НЕ включаем — иначе любой сдвиг вёрстки даст ложную «новую» находку.

`SourceLocation` достаём из `SemanticsNode` через `debugCreator` (в debug/test-сборках Flutter хранит `Element`, по которому можно поднять `DiagnosticsNode` и место создания виджета).

## 3. Методы детекции (три класса проверок)

| Метод | Что нужно | Какие правила |
|---|---|---|
| **TREE** — обход `SemanticsNode` | только `tester.getSemantics()` / `ensureSemantics()` | R1, R2, R6, R7, R8, R10, R11, R12 |
| **RASTER** — растеризация + сэмплирование пикселей | golden-инфраструктура (`tester.pumpAndSettle` + `RepaintBoundary.toImage`) | R3 (контраст) |
| **TEXTSCALE** — повторный pump при увеличенном шрифте | `MediaQuery` override + перехват overflow | R5 |

TREE-правила — дешёвые и детерминированные, делаем их первыми. RASTER и TEXTSCALE дороже, но именно они закрывают то, чего нет ни у кого.

### R1 — Интерактивный элемент без доступного имени
**WCAG 4.1.2 Name, Role, Value (A) · EN 301 549 §11.5.2.5**

Скринридер озвучит такой элемент как «кнопка» без указания, что она делает. Самая частая и самая критичная находка.

**Детекция (TREE):** узел имеет `SemanticsAction.tap` ИЛИ флаг `isButton`/`isLink`, при этом `label.trim().isEmpty` И `tooltip` пуст И среди дочерних узлов нет текстового, дающего имя.

```dart
bool violatesR1(SemanticsNode n) {
  final d = n.getSemanticsData();
  final interactive = d.hasAction(SemanticsAction.tap) ||
      d.hasFlag(SemanticsFlag.isButton) ||
      d.hasFlag(SemanticsFlag.isLink);
  final named = d.label.trim().isNotEmpty || d.tooltip.trim().isNotEmpty;
  return interactive && !named && !_hasNamingChild(n);
}
```

**Ложные срабатывания:** обёртки-`GestureDetector` чисто для жестов (свайпы), декоративные тап-зоны. → механизм `// ignore: a11y/interactive-name` или `Semantics(excludeSemantics: ...)`.

**Fix:** обернуть в `Semantics(label: 'Оплатить', button: true, child: ...)` либо использовать `IconButton(tooltip: ...)`.

---

### R2 — Изображение/иконка без текстовой альтернативы
**WCAG 1.1.1 Non-text Content (A) · EN 301 549 §11.5.2.5**

**Детекция (TREE):** флаг `isImage` установлен И `label.trim().isEmpty` И узел не помечен декоративным. Отдельно ловим частый кейс: `IconButton`/`Icon` как единственный значимый ребёнок интерактивного узла без метки (пересекается с R1, но репортим по более точному критерию).

**Ложные срабатывания:** чисто декоративные изображения, которые корректно скрыты. Если разработчик осознанно скрыл картинку через `ExcludeSemantics` — нарушения нет (это и есть правильное «decorative»). Поэтому проверяем именно «видимый для семантики image без label».

**Fix:** `Image(..., semanticLabel: 'График роста выручки')` или, для декора, `ExcludeSemantics(child: Image(...))`.

---

### R3 — Недостаточный контраст текста
**WCAG 1.4.3 Contrast (Minimum) (AA) · EN 301 549 §11.5.2.4**

Порог: обычный текст ≥ 4.5:1, крупный (≥ 24px, либо ≥ 18.66px bold) ≥ 3:1.

**Детекция (RASTER):** растеризуем поддерево виджета в изображение, для каждого текстового узла берём его `bounds`, сэмплируем пиксели глифов (тёмные/яркие кластеры) против усреднённого фона в той же области, считаем относительную яркость по формуле WCAG и контраст `(L1 + 0.05) / (L2 + 0.05)`.

```dart
double contrastRatio(Color fg, Color bg) {
  final l1 = _relativeLuminance(fg);
  final l2 = _relativeLuminance(bg);
  final hi = max(l1, l2), lo = min(l1, l2);
  return (hi + 0.05) / (lo + 0.05);
}
```

**Ложные срабатывания:** текст поверх градиента/фото, полупрозрачные слои, **disabled-контролы (WCAG 1.4.3 их исключает — обязательно отфильтровать по флагу `isEnabled == false`)**. Для borderline-значений (4.3–4.5) понижаем до `warning`.

**Fix:** скорректировать цвет темы; в отчёте предлагаем ближайший проходящий оттенок.

---

### R4 — Слишком маленькая тап-зона
**WCAG 2.5.8 Target Size (Minimum) (AA, WCAG 2.2, 24×24) · платформенный гайд: Material 48×48, iOS 44×44**

**Детекция (TREE):** узел имеет tap-action; считаем глобальный размер `bounds`; флагуем, если меньшая сторона ниже порога (конфигурируемо: строгий WCAG 2.2 = 24, рекомендуемый платформенный = 48/44).

**Ложные срабатывания:** у 2.5.8 есть исключения — inline-ссылки в тексте, «essential» элементы, достаточный spacing вокруг. Поэтому severity = `warning`, и учитываем зазор до соседних таргетов.

**Fix:** увеличить область через `padding`/`SizedBox`/`MaterialTapTargetSize.padded`.

---

### R5 — Overflow вёрстки при увеличенном шрифте
**WCAG 1.4.4 Resize Text (AA) + 1.4.10 Reflow (AA)**

Киллер-фича: на Flutter крупный системный шрифт регулярно ломает `Row`/`Column` (`RenderFlex overflowed`), и веб-инструменты это в принципе не видят.

**Детекция (TEXTSCALE):** повторно прогоняем экран при `textScaler` = 1.3 и 2.0, перехватываем `FlutterError` про overflow и детектим клиппинг текста (когда `paintBounds` текста шире контейнера).

```dart
await tester.pumpWidget(
  MediaQuery(
    data: MediaQuery.of(ctx).copyWith(textScaler: const TextScaler.linear(2.0)),
    child: screen,
  ),
);
// собрать перехваченные overflow-ошибки этого pump
```

**Fix:** заменить фиксированные высоты на гибкие, обернуть в `Flexible`/`FittedBox`, разрешить перенос строк.

---

### R6 — Дублирующиеся/неоднозначные имена
**WCAG 2.4.6 Headings and Labels (AA)**

**Детекция (TREE, эвристика):** на одном экране несколько интерактивных узлов с идентичной меткой, причём метка либо генерик («ещё», «открыть»), либо одинаковая при разном действии (две кнопки «Удалить» без уточнения, что именно).

**Fix:** уточнить метки («Удалить фото профиля» vs «Удалить аккаунт»).

---

### R7 — Плейсхолдер вместо осмысленного имени
**WCAG 2.4.6 / 4.1.2 · EN 301 549 §11.5.2.5**

**Детекция (TREE):** `label` матчится по деноллисту генериков и заглушек: `button`, `image`, `icon`, `label`, `text`, `todo`, `untitled`, `flutter`, одиночные эмодзи и т.п. (case-insensitive, с локализованными вариантами).

**Fix:** задать содержательную метку.

---

### R8 — Поле ввода/контрол без метки
**WCAG 1.3.1 Info and Relationships (A) + 3.3.2 Labels or Instructions (A) + 4.1.2**

**Детекция (TREE):** флаг `isTextField`, ЛИБО `hasCheckedState`/`hasToggledState` (checkbox/switch/radio), при `label.trim().isEmpty` и без связанного узла-метки рядом. Учитываем, что `hintText` (placeholder) НЕ считается меткой — это частая ошибка.

**Fix:** `TextField(decoration: InputDecoration(labelText: ...))` или обернуть в `Semantics(label: ...)`; для чекбоксов — связать с подписью.

---

### R9 — Отсутствует семантика заголовков
**WCAG 1.3.1 + 2.4.6**

**Детекция (TREE, эвристика):** на экране есть визуально «заголовочный» текст (крупный кегль + bold, по `TextStyle`), но ни один узел не несёт флаг `isHeader`. Эвристика шумная — поэтому `warning` и явная пометка `heuristic` в отчёте, чтобы её можно было массово приглушить.

**Fix:** `Semantics(header: true, child: Text('Раздел'))`.

---

### R10 — Состояние элемента не озвучивается
**WCAG 4.1.2 Name, Role, Value — часть «Value/State» (A) · EN 301 549 §11.5.2.5**

**Детекция (TREE, эвристика):** кастомный контрол (собран из `GestureDetector` с tap-action) визуально передаёт состояние (выбран/включён — например, по подсветке), но узел не имеет `hasCheckedState`/`isSelected`/`hasToggledState`. Полностью обобщить трудно; в MVP стартуем с кастомных табов/чипов/тогглов.

**Fix:** прокинуть состояние в `Semantics(selected: true, ...)` / `toggled`.

---

### R11 — Нелогичный порядок фокуса
**WCAG 2.4.3 Focus Order (A)**

**Детекция (TREE, эвристика):** сравниваем `childrenInTraversalOrder` с визуальным порядком чтения (сортировка по глобальному `bounds`: сверху-вниз, слева-направо для LTR). Флагуем крупные инверсии (например, фокус прыгает снизу экрана наверх).

**Fix:** упорядочить через `FocusTraversalGroup` / `Semantics(sortKey: OrdinalSortKey(...))`.

---

### R12 — Ловушка фокуса / недостижимый интерактивный элемент
**WCAG 2.1.1 Keyboard (A) + 2.4.3 (A)**

**Детекция (TREE):** интерактивный узел (есть tap-action), который при этом либо помечен `isHidden`, оставаясь тапабельным, либо накрыт `ExcludeSemantics` и потому не попадает в порядок обхода, хотя визуально присутствует и кликается. Такой элемент существует для зрячего пользователя, но недостижим для скринридера/клавиатуры.

**Fix:** убрать ошибочный `ExcludeSemantics`/`isHidden`, вернуть элемент в дерево семантики.

## 5. Чего этот набор НЕ покрывает (раздел честности)

Это нужно прямо писать в README и в отчёте, чтобы не вводить пользователя в юридическое заблуждение:

- **1.4.1 Информация только цветом** — требует анализа смысла, автоматом почти не ловится.
- **1.2.x Медиа (субтитры, аудиоописание)** — вне автоматической проверки.
- **3.3.x Содержательность сообщений об ошибках** — требует человека.
- **Осмысленность alt-текста** — мы видим, что метка есть; полезна она или «image123» по смыслу — нет.
- **Реальный тест со скринридером** (TalkBack/VoiceOver) — незаменим; инструмент его дополняет, а не заменяет.

Формула позиционирования: **«ловим всё, что машинно проверяемо, и даём структурированный чеклист по остальному»**.

## 6. Инфраструктура вокруг правил (вторая итерация, после ядра)

- **Baseline-гейт:** файл `.a11y/baseline.json` со списком `fingerprint` принятых находок. CI падает только на новых. Команда `a11y_audit baseline --update` пересохраняет.
- **Вывод в SARIF:** тогда GitHub/GitLab рисуют аннотации прямо в diff PR. Плюс JSON (машинам) и HTML (людям).
- **Транскрипт скринридера:** отдельный режим — печатает в лог CI последовательность того, что озвучит TalkBack/VoiceOver при обходе экрана в traversal-порядке. Дифференциатор, которого нет ни у кого.
- **Conformance-отчёт + черновик EU accessibility statement** (по модели Implementing Decision 2018/1523) с трассировкой находок к пунктам стандарта — это уже платный слой.

---

## 7. Рекомендуемый порядок разработки

1. **Каркас + TREE-движок + R1, R2, R7, R8.** Это «дешёвые», детерминированные, самые частые находки. Уже на этом этапе пакет полезен и его можно публиковать.
2. **R4, R12, R6.** Геометрия и достижимость.
3. **R5 (TEXTSCALE).** Flutter-специфичный killer-feature, отдельный pump-режим.
4. **R3 (RASTER).** Контраст — самая инженерно-ёмкая часть (растеризация + сэмплинг).
5. **R9, R10, R11 (эвристики).** Выпускать с явной пометкой `heuristic` и удобным глушением.
6. **CI-обвязка:** baseline-гейт → SARIF → транскрипт скринридера.
7. **Платный слой:** дашборд, conformance-отчёт, accessibility statement.

После шага 1 у тебя уже публикуемый OSS-пакет; всё остальное наращивается итерациями без переписывания ядра.
